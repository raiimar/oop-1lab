# Проверка лабораторной работы по ООП

## Краткий итог
Работа в целом решает поставленную задачу: есть базовый абстрактный класс `Shape`, полиморфизм по `getPerimeter()` и `toString()`, хранение в общей коллекции, обработка ошибок через исключения.

Ниже — замечания по качеству ООП, читаемости, именам и декомпозиции.

---

## 1. Соответствие принципам ООП

### Что сделано хорошо
- Применено наследование: `Circle`, `Rectangle`, `Triangle` наследуются от `Shape`.
- Базовый класс абстрактный, что уместно для предметной области.
- Используется полиморфизм и единый контейнер `std::vector<std::shared_ptr<Shape>>`.
- В конструкторах фигур проверяются инварианты (невырожденность/корректность параметров).

### Что можно улучшить
1. **Нарушение SRP (Single Responsibility) в `ShapeAdder`**
   - Класс одновременно:
     - ведёт диалог с пользователем;
     - парсит ввод;
     - знает правила валидации ввода;
     - создаёт доменные объекты.
   - Лучше разделить:
     - `ShapeFactory` (создание фигур из готовых параметров);
     - `ShapeInputController` (сбор данных с консоли);
     - `ShapeManager` (операции над коллекцией).

2. **Дублирование бизнес-логики: `Commands` и `ShapeManager`**
   - Сейчас есть два класса с почти одинаковым поведением. Это повышает риск расхождения логики и усложняет поддержку.
   - Нужно оставить один класс (предпочтительно `ShapeManager`) и удалить `Commands`.

3. **Слабая инкапсуляция в `Point`**
   - Поля `x`/`y` публичные. Для модели значения лучше сделать их приватными и задавать через конструктор.
   - Это предотвратит неконтролируемые изменения состояния.

## 2. Читаемость и стиль

1. **Имена переменных**
   - `ShapeManager command` в `main.cpp` и поле `command` в `Menu`/`ShapeAdder` не отражают смысл.
   - Рекомендуется переименовать в `shapeManager`/`manager`.

2. **Пустые ветки `if` в `Input`**
   - Конструкции вида `if (std::cin >> value) { } else { ... }` ухудшают читаемость.
   - Лучше использовать раннее присваивание флага или отдельную функцию чтения.

3. **Макросы вместо `constexpr`**
   - `#define eps`, `#define half`, `#define double_pi` лучше заменить на `constexpr double`.
   - Это безопаснее и современнее для C++.

4. **Платформозависимость**
   - `#include <windows.h>` в `output.cpp` делает проект непереносимым.
   - Лучше обернуть в `#ifdef _WIN32` или вынести настройку кодировки в отдельный адаптер.

5. **Опечатка в include**
   - В `point.cpp` подключается `"Point.h"`, а файл называется `point.h`.
   - На Linux/macOS это приведёт к ошибке сборки.

## 3. Декомпозиция функций и ограничение “1 return / до 45 строк”

### По текущему коду
- Большинство функций короче 45 строк — это хорошо.
- Есть функции с несколькими `return` (например, `Menu::run`, `ShapeAdder::addShape`), что нарушает ваше правило.

### Рекомендация по паттерну
Используйте локальную переменную результата и один `return` в конце:

```cpp
bool ok = true;
if (!input.readInt(choice)) {
    ok = false;
}

if (ok) {
    // основная логика
}

return ok;
```

Для длинных функций (`run`, добавление фигур) — разбивать на небольшие шаги:
- `readMenuChoice(...)`
- `handleChoice(...)`
- `executeAction(...)`

Это улучшит и читаемость, и тестируемость.

## 4. Соответствие названий реальному применению

### Удачно
- `Shape`, `Circle`, `Rectangle`, `Triangle`, `getPerimeter` — предметно и понятно.

### Неудачно / спорно
- `Commands` — слишком общее название и дублирует `ShapeManager`.
- `command` как объект-менеджер фигур — вводит в заблуждение.

### Предлагаемое переименование
- `command` -> `shapeManager`;
- `ShapeAdder` -> `ShapeCreationService` (или `ShapeConsoleCreator`, если класс именно про ввод из консоли).

---

## 5. Конкретный план исправлений

1. Удалить `Commands` и использовать только `ShapeManager`.
2. Переименовать переменные `command` в `shapeManager`.
3. Исправить include в `point.cpp` на `#include "point.h"`.
4. Сделать `Point` неизменяемым value-object:
   - приватные поля;
   - конструктор `Point(double x, double y)`;
   - только `getX()/getY()`.
5. Заменить `#define` на `constexpr` в `circle.cpp` и `triangle.cpp`.
6. В `Output` добавить условную компиляцию для Windows API.
7. В `Menu::run` и `ShapeAdder::addShape` перейти на схему с одним `return`.
